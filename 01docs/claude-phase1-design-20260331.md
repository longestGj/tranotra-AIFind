# Design: Tranotra Leads Phase 1 — LinkedIn数据采集 + 去重 + Web展示

**生成日期**: 2026-03-31  
**项目**: tranotra-leads  
**负责人**: Tranotra Chemical  
**状态**: DRAFT  
**模式**: Startup  
**阶段**: Phase 1 (MVP)

---

## 1. 问题陈述

Tranotra Chemical 是一家增塑剂出口商，目标市场在东南亚和中东。

**核心痛点**：
- 每天手动在LinkedIn上搜索目标客户，耗时30-45分钟找10个客户
- 成功概率只有10%（需要筛选大量不符合的公司）
- 找到的数据用Excel手动记录，难以去重和分析
- 每周花20小时在搜索和记录上

**目标**：
构建自动化系统，用Gemini搜索目标市场的潜在买家，自动去重，存入数据库，通过Web界面查看和管理。让人工时间从20小时/周降到5小时/周。

---

## 2. 需求证据

**定量数据**：
- 当前每天必须做这项工作（日常高频重复）
- 成功率10%（需要大量筛选）
- 时间从20小时降到5小时就很满足（75%的时间节省）
- 愿意投3周时间开发

**定性证据**：
- 这是一个缓慢的行业，找到一个好客户可以长期合作
- 真实痛点：不是"找客户"本身，而是"记录、去重、分析客户数据"
- 用户已经有Gemini和Apollo的API账号，说明对自动化工具的认可

---

## 3. 核心假设 & 约束

### 已验证的假设：
1. ✅ Gemini能从搜索结果中提取LinkedIn URL（用户已验证可行）
2. ✅ Gemini返回结构化数据（CSV格式，16列）
3. ✅ 接受抓取失败（如果LinkedIn上找不到官网，先不找）
4. ✅ 接受AI推断的不完美性（去重可能有漏网之鱼，可接受）
5. ✅ 分阶段开发（Phase 1只做LinkedIn数据采集，不涉及官网分析）

### 核心约束：
- **本地运行**，不需要云端定时任务
- **Web界面**而非CLI（中等复杂度：表单+表格+历史+排序+导出）
- **SQLite数据库**本地存储
- **LinkedIn URL为去重唯一键**（同一LinkedIn页面只保留一条记录）
- **Flask技术栈**（快速、够用）

---

## 4. 用户故事

**作为** Tranotra的运营者，  
**我希望** 在Web界面输入"国家 + 搜索词"，系统自动搜索该国家相关公司并去重，  
**以便** 我可以在10分钟内看到一个干净的LinkedIn公司列表，而不是花30分钟手动LinkedIn搜索。

**验收标准**：
- 输入"Vietnam" + "PVC manufacturer"，得到10-20家去重后的公司列表
- 列表包含：公司名、国家、搜索词、描述、LinkedIn URL、官网
- 可以按评分/国家排序，导出CSV
- 搜索历史可查看，不会重复搜

## 5. Phase 1 系统设计

### 5.1 技术栈

| 组件 | 选择 | 理由 |
|------|------|------|
| 后端框架 | Flask | 轻量、快速开发、3周内足够 |
| 数据库 | SQLite | 本地存储、无运维、足够容纳10k+公司 |
| 前端 | HTML + Bootstrap + Vanilla JS | 简洁、无依赖、快速展示 |
| AI能力 | Gemini API | 搜索 + 数据提取 |
| 部署 | 本地运行 | 用户电脑上 `python main.py` |

### 5.2 数据库设计（Phase 1简化版）

**主表：`companies`**

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | INTEGER | PK, AUTO_INCREMENT | 自增主键 |
| `name` | TEXT | NOT NULL | 公司名称 |
| `country` | TEXT | NOT NULL | 国家 |
| `source_query` | TEXT | NOT NULL | 搜索时使用的关键词 |
| `description` | TEXT | | Gemini返回的公司描述 |
| `linkedin_url` | TEXT | UNIQUE | LinkedIn公司页面URL（去重键） |
| `website` | TEXT | | 官网（从LinkedIn推断，可为空） |
| `created_at` | DATETIME | DEFAULT CURRENT_TIMESTAMP | 创建时间 |
| `updated_at` | DATETIME | DEFAULT CURRENT_TIMESTAMP | 更新时间 |

**辅表：`search_history`**

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | INTEGER | PK | 自增主键 |
| `country` | TEXT | NOT NULL | 搜索的国家 |
| `query` | TEXT | NOT NULL | 搜索关键词 |
| `result_count` | INTEGER | | 本次搜索返回的公司数 |
| `created_at` | DATETIME | | 搜索时间 |

### 5.3 系统流程

```
用户在Web界面输入：国家 + 关键词
        ↓
Flask后端接收请求，记录到search_history
        ↓
调用Gemini Grounding Search API
        ↓
Gemini返回CSV（16列数据）
        ↓
提取关键字段：name, country, description, linkedin_url, website
        ↓
检查linkedin_url是否已在数据库中（去重）
  - 如果存在：跳过
  - 如果不存在：插入companies表
        ↓
查询数据库，返回本次搜索的所有新增公司
        ↓
Web前端展示表格：公司名 | 国家 | LinkedIn | 官网 | 搜索词
```

## 6. 开发里程碑 & 时间分解

| 里程碑 | 包含内容 | 时间 | 验收标准 |
|--------|--------|------|---------|
| **M1** | 数据库初始化 + Flask框架搭建 + Gemini集成 | 3-4天 | `python main.py` 能跑，能调用Gemini，返回数据 |
| **M2** | CSV解析 + 去重逻辑 + 数据入库 | 3-4天 | 搜索一次Vietnam/PVC，数据正确入库，去重正常 |
| **M3** | Web前端：搜索表单 + 结果表格 + 排序导出 | 3-4天 | Web界面可用，能搜索、排序、导出CSV |
| **M4** | 搜索历史 + 完整测试 + 部署文档 | 2-3天 | 所有功能完成，写好README，用户可独立运行 |

**总计**：2周（假设每天6-8小时开发）

## 7. 可能的风险 & 缓解方案

| 风险 | 可能性 | 影响 | 缓解方案 |
|------|--------|------|---------|
| Gemini搜索质量不达预期（太多无关结果） | 中 | 去重和分析工作量增加 | 优化搜索词模板，迭代提示词 |
| LinkedIn URL解析失败（CSV格式变化） | 低 | 无法去重 | 预留2-3小时debug |
| 官网无法从LinkedIn推断（LinkedIn页面不完整） | 中 | 数据不完整，但可接受 | 接受"官网：未找到"，进入Phase 2再补 |
| 数据库性能（10k+公司的查询） | 低 | 表格加载慢 | SQLite足够，需要时加索引 |
